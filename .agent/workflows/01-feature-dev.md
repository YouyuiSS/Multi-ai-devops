# 独立产品新功能开发标准流 (Feature Development Workflow)

> 此工作流用于替代臃肿的 OpenSpec 框架，以极客的轻量级方式强制 AI “先想后写”。

## 步骤 1: 需求阅读与推演 (Proposal & Design)
请精读 `docs/01_产品设计/` 和 `docs/03_技术设计/` 中与本次需求相关的文件。
分析本次新增功能对现有系统的影响，并输出一个简短的改动评估（包括需要修改的数据库表、API、核心组件）。
**等待用户确认（Approve）后，再进入下一步。**

## 步骤 2: 任务拆解 (Task Breakdown)
根据推演结果，生成一个清晰的 `TODO_当前功能名.md` 任务清单（请存放到临时目录或根目录）。
任务必须拆解到“单个函数/文件/组件”级别。

## 步骤 3: 代码实现 (Implementation)
遵循 `.agent/rules/` 中的全局规范，开始按照 `TODO` 文件中的步骤逐一编写代码。
写完一个模块，在 TODO 文件中将其标记为 `[x]`。
**如遇报错或不确定的边界条件，优先输出日志和假设，不要擅自修改无关的底层核心代码。**

## 步骤 4: 自测与审查 (Self-Review)
代码写完后，自己阅读一遍差异（`git diff`）。
确认没有遗漏日志打印、异常捕获和权限校验。
// turbo (如果项目支持自动运行测试，可以在这里静默执行 npm test 或 mvn test)

## 步骤 5: 总结与交接 (Handoff)
阅读 `.agent/rules/handoff_template.md`，按其格式输出你本次开发的总结。
清理工作区（删除临时的 TODO 文件）。
