# 基于 Antigravity 的独立产品测试指南

在产品开发完成核心逻辑后，保障其稳定性是独立开发者最容易忽略，但却最关键的一环。
编写繁琐的单元测试往往让人望而却步，但在 Antigravity 中，这一切可以被极大程度地自动化和简化。

---

## 一、 测试阶段：Antigravity 能做什么？

Antigravity 擅长理解你的业务逻辑上下文，这让它在测试用例的生成上有极大的优势：

### 1. 自动生成高质量的单元测试 (Unit Testing)
当你写完一个核心业务类（例如后端的 Service 接口或前端的 Hook 逻辑），你可以直接在当前文件上下文中要求 Antigravity：
*   **生成边界用例**：让它不仅仅测“主干道正常流程”，还要主动补充入参为空、各种极值、未捕捉业务异常抛出等非预期场景的断言（Assertions）。
*   **Mock 外部依赖**：如果你使用了数据库读写、第三方 API 请求，Antigravity 能够根据你的技术栈自动生成合适的拦截与模拟代码（例如 Jest/Vitest 的 `vi.mock` 或 Java 的 `@Mock`），保证单元测试的纯粹性且不需要真实数据库即可极速执行。
*   **指令示例**：
    > “请针对当前我打开的 `AuthService.ts` 编写 Jest 单元测试。覆盖全部 public 暴露的方法，尤其是密码错误、账号锁定时的边界情况，并使用 ts-mock 拦截底层所有的数据库 ORM 查询。”

---

## 二、 真正的次世代端到端 (E2E) 测试：Browser Subagent

你说的没错，传统的“让 AI 写 Cypress 脚本再依靠运行器跑”的方式实在过于低效，也经不起 UI 的细微变动（一加一个 div 层级脚本就崩了）。

作为新一代 Agent，**Antigravity 拥有原生的 Browser Subagent (浏览器子代理) 能力**。它不需要生成脚本再去死板地执行，它自己长着“眼睛”和“手”！

### 1. 什么是 Antigravity 浏览器代理测试？
当您的本地开发服务器（如 `localhost:3000`）运行起来后，您可以直接给 Antigravity 下达**意图驱动（Intent-driven）**的测试指令。
Antigravity 会启动一个后台的隐匿浏览器（带着真实的渲染引擎），它能像真人一样：
*   **去识别屏幕上的按钮和文字**（无需死板依赖 DOM 的 id 或是 class 层次）。
*   **去执行诸如点击、滚动、输入甚至拖拽的动作**。
*   同时，它的所有行为都会被**自动录制为 WebP 格式的视频动画**，以供你审查复盘。

### 2. 实操演示：如何发起一次自动化验收

你无需学习任何自动化框架语法。打开终端，用 `npm run dev` 把项目跑起来，然后对 Antigravity 抛出这样的指令：

> “请利用 `browser_subagent` 帮我测试一下整个购买链路。
> 1. 打开 `http://localhost:3000/shop`。
> 2. 我们设定的测试主线是：搜索「无线耳机」，点击第一个进入详情页，将它加购，然后点击右上角购物车按钮去算钱。
> 3. 请你帮我顺着点一遍，测试下中途有没有什么 UI 加载报错或是控制台飘红。
> 4. 测试录像麻烦存为 `purchase_flow_test` 备查。”

在收到指令后，Antigravity 会召唤它的浏览器分身。
你会看到它回复你类似于：_“好的，我将触发子代理为您遍历……”_ 
一段时间后，它会带着**录屏文件**和**通过/失败的最终结论**回到聊天窗口向你汇报。测试通过，就是真真实实地在浏览器里跑痛了。

## 三、 高阶应用：全自动伴随式测试 (测试左移固化)

**问题：** 每次改完代码都要手动唤起测试，极其反人性，最后肯定无法坚持。
**解法：** 利用 Antigravity 的 **Rules (规则匹配引擎)** 将测试彻底接管并固化到保存动作之后，实现**后端改完自动验单测，前端改完自动跑录屏**。

### 1. 后端接口防劣化：自动运行并补全代码层级单元测试
利用 Glob 路径匹配规则，让 AI 在监测到核心后端逻辑更新（如 Service/Controller）层变动时，静默拉起测试自动机。

**实操步骤：**
在项目根目录创建并配置 `.agent/rules/backend-auto-test.md`：
```markdown
---
name: Backend Auto Testing Guard
description: Auto-trigger unit tests when backend logic changes
match: apps/api/**/*.ts # (或者是 src/main/java/**/*.java)
mode: auto
---

**触发强制校验流程 (Test Guard)：**
1. 我只要修改了你匹配到的代码文件，你必须在回答我之前，自动运行项目根目录下的测试脚本 (例如 `npm run test -- <受影响的文件名>`)。
2. 你要仔细分析报错的终端输出：
   - 没过：不需要等我同意，你直接基于报错堆栈继续修改代码直到 Pass。
   - 用例缺失：如果你发现是我新增了一个 public 的方法导致覆盖率下降，请你主动生成并把测试用例补充进对应的 `xxx.spec.ts` 文件中。
3. 最终只给我看一个大获全胜的 `[测试全量通过]` 总结即可。
```

### 2. 前端变更防崩溃：保存即触发 Browser Subagent 验收
既然我们在上一节拥有了不需要脚本的 Browser Subagent，我们就可以利用规则把它挂载到前端页面的保存动作中！

**实操步骤：**
不仅要配置规则，还要给它写好明确的测试剧本（SOP）。
创建并配置 `.agent/rules/frontend-auto-qa.md`：
```markdown
---
name: Frontend Auto QA with Browser
description: Auto-trigger Browser Subagent to perform visual regression QA
match: apps/web/src/pages/**/*.tsx # (也就是你的页面/视图路由组件目录)
mode: auto
---

**触发强制视觉验收流程 (Visual QA Guard)：**
一旦我在跟你结对编程时让当前这一页的代码发生了任何逻辑或渲染结构的变动，你必须完成以下闭环操作，并且不需要我的二次确认：
1. 请你主动调用 `browser_subagent` 工具，从 `http://localhost:3000` 进入我当下正在修改的这个页面路由。
2. 找到本目录（`apps/web/e2e-docs`）下与我当下修改的文件同名的 Markdown 交互文档（这相当于你测试该页面的 SOP 剧本）。
3. 按照剧本的描述，让 Browser Subagent 真人模拟把核心链路点一遍。
4. 如果跑通了，就在聊天窗口丢一个截屏；如果弹窗、红字或者点不动了，帮我把刚才写的代码复原回去并排查问题。
```

## 结语

基于上述配置，Antigravity 不再是你敲击回车才动一下的传统工具，它变成了一个**结对编程里的全能副驾**。
它一直在后台盯着你的后背：你改动后端的接口数据格式，它扭头就把单元测试修成绿灯；你前脚刚把“登录”按钮重构成了一个更漂亮的组件，它后脚就立刻派出了浏览器分身替你走完鉴权流程并向你点头致意。
